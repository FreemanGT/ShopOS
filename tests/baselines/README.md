# Regression baselines (Wave 0.4)

These files are frozen snapshots of the public surface of shopos-core and
shopos-digital as of v1.10.15 (2026-04-29). Wave 1+ PRs must not silently
mutate them — if a PR's changes alter any baseline, the diff appears in code
review and the author must justify the change in the PR description.

## Files

| File | What it captures | Source |
|---|---|---|
| `tests/baseline-hooks.txt` | Every `apply_filters(...)` and `do_action(...)` call site in `shopos-core/src` (file:line:line-text). | static grep |
| `tests/baseline-rest.txt` | Every `register_rest_route(...)` call site in both packages. Currently empty per `/docs/decisions-2026-04-28.md` §4.4. | static grep |
| `tests/baseline-cli.txt` | Every `WP_CLI::add_command(...)` call site in both packages. | static grep |
| `tests/baseline-options-declared.txt` | Every prefix-matching identifier (`shopos_[a-zA-Z0-9_]+`) found inside any **real** PHP string literal across both packages, deduped and sorted. | PHP tokenizer (see below) |
| `tests/baseline-options-legacy.txt` | Same, for `shopos_[a-zA-Z0-9_]+`. Per `/docs/decisions-2026-04-28.md` §4.5, the legacy `shopos_*` surface must keep working until a sunset is explicitly approved. | PHP tokenizer |
| `tests/baseline-options.txt` | **Live snapshot.** `wp option list --search='shopos_*' --format=csv` against a clean WP install with both plugins activated. Captured manually; not regenerated by CI. | live `wp option list` |

The two `*-options-declared` / `*-options-legacy` files capture **identifiers**, not just option keys. The same prefix string is also used for hook names, AJAX action slugs, transient keys, capability slugs, and admin URLs — all of which Wave 1+ also must not silently rename. A broader baseline is more honest about what the code actually exposes.

Identifier extraction uses [tools/extract-identifiers.php](../../tools/extract-identifiers.php), which walks each PHP file via `token_get_all()` and only inspects `T_CONSTANT_ENCAPSED_STRING` tokens. **Comments and docblocks are never matched**, so a future PR that mentions `'shopos_foo'` in a `//` line comment or a `@param` docblock will not cause spurious baseline diffs. The tokenizer also avoids substring-matching false positives because it only looks at real string literals — but within those literals, `preg_match_all` runs unanchored, so an identifier that appears as a substring of a larger literal (e.g. `shopos_vs` inside an admin URL) is still captured.

## Regenerating the static baselines

```bash
bash tools/capture-baselines.sh
```

The script is idempotent: running it twice produces zero diff. It exports `LC_ALL=C` so byte-order sort is identical on macOS (BSD coreutils) and Linux (GNU coreutils). Tested on macOS 14 / bash 3.2 and Ubuntu 22.04 / bash 5.

The script invokes `php` for identifier extraction. Override the binary by exporting `PHP_BIN=/path/to/php` — useful on macOS where the system `php` may be 8.5 (outside PHPUnit 10's range) and you want the same 8.3 binary CI uses.

If you regenerate locally without `LC_ALL=C` set in your shell, the script still locks it internally — so output is deterministic regardless of your locale.

## Regenerating the live options baseline

The live baseline requires WP-CLI and a real WordPress install with both plugins activated. Capture it manually after material changes (e.g., a new release that adds option keys). Use this exact pipeline so the result is comparable to the committed file:

```bash
# On a WP install with shopos-core + shopos-digital activated:
{
    wp option list --search='shopos_*' --format=csv 2>/dev/null | head -n 1
    wp option list --search='shopos_*' --format=csv 2>/dev/null | tail -n +2 | LC_ALL=C sort
} > /path/to/repo/tests/baseline-options.txt
```

Three details matter:

1. **`2>/dev/null`** — suppresses environment-specific PHP startup warnings (e.g. missing `imagick.so` on Local-by-Flywheel) that would otherwise pollute the file and produce machine-dependent diffs.
2. **Header preserved on line 1** — the CSV header (`option_name,option_value`) is split out before `sort` so it doesn't end up alphabetically buried among the data rows.
3. **`LC_ALL=C sort`** — same byte-order locking as the static baselines, so a refresh on Linux matches a refresh on macOS.

This file is **not** asserted by CI — it's a human-curated reference snapshot. Wave 2.2 (VariationSwatches migration) will diff against it manually as part of its rollout drill.

The current snapshot is intentionally minimal (3 rows): a fresh activation only writes `shopos_core_db_version`, `shopos_core_modules`, and `shopos_core_productfeed_last_generated`. Most `shopos_*` identifiers in the static `baseline-options-declared.txt` are hook names, AJAX action slugs, transients, or nonces — not WP options. The two files complement each other rather than overlap.

## How CI enforces this

`tests/BaselinesIntegrityTest.php` re-runs `tools/capture-baselines.sh` into a temp directory and asserts the output matches the committed static baselines byte-for-byte. A failing test means the source code's hook/REST/CLI surface or its `shopos_*` / `shopos_*` identifiers drifted without the baseline file being updated in the same PR.

To resolve a failing baseline test:

1. Run `bash tools/capture-baselines.sh` locally.
2. Inspect the diff against the committed baseline.
3. If the change is intentional, commit the updated baseline alongside the source change and explain it in the PR description.
4. If the change is unintentional, fix the source until the baseline matches again.

## Rationale

Per `/docs/roadmap.md` Wave 0.4: "These are the snapshots Wave 1+ must not break." The intent is to make accidental renames, deletions, or REST/CLI surface drift visible in code review rather than discovered post-merge.
