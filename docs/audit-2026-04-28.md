# ShopOS Plugin Suite — Feature & Customization Gap Audit

## Context

This is a **read-only audit** of the ShopOS plugin suite at `/Users/freemansmain/Ai Projects/ShopOS/`, comprising:

- **shopos-core** v1.10.12 — WooCommerce feature plugin (9 modules)
- **shopos-digital** v1.7.3 — performance/optimization plugin (12 sub-modules)
- **shopos-theme** v1.10.8 — Hello Elementor child theme

Goal: surface missing features, hardcoded design tokens that should be customizable, hardcoded behaviors that should be configurable, and developer extension gaps. **No code changes are produced** — output is a written report only.

The user is a senior product strategist auditing the plugin's commercial readiness against best-in-class WooCommerce competitors (e.g. WPC Variation Swatches, FooEvents, MailPoet, SiteGround Optimizer, WP Rocket).

---

## 1. Inventory Summary

| Package | Version | Surface |
|---|---|---|
| shopos-core | 1.10.12 | 9 modules: CategorySlider, ProductSlider (Elementor widgets), CheapestDefaultVariation, InfiniteScroll, MyAccount (CSS-only), ProductFeed (XML feed at `/product-feed`), RestockNotify (legacy menu + shortcode + DB table `shopos_restock_subscribers`), VariableStockFix, VariationSwatches (legacy buy-box shortcode `[shopos_buy_box]`) |

**Settings architecture**: declarative schema in `Settings_Hub` auto-renders fields (`text|textarea|checkbox|select|color|number|email|url`); options keyed `shopos_core_<module_id>_<setting>`; capability `manage_woocommerce`. **No REST API endpoints across the entire suite**. **Logging** via `ShopOS\Core\Core\Logger` (option-backed, last 100 entries).

### C. Missing Customizations — Functional

#### C1. CategorySlider / ProductSlider
- No **autoplay**, **loop**, **pagination dots** controls (only arrows + drag).
- No **infinite carousel** mode.
- No **mobile-only / desktop-only** display toggle on widget.
- No **schema-org markup** option for Product Slider (would help SEO).
- `direction` (auto/ltr/rtl) ([Widget.php controls](shopos-core/src/Modules/CategorySlider/Widget.php)) is per-instance — should optionally be force-overridable per-locale.
- No **filter `shopos_core/category_slider/query_args`** — query is built inline ([Widget.php:456-462](shopos-core/src/Modules/CategorySlider/Widget.php#L456-L462)).

#### C2. CheapestDefaultVariation — [Module.php:65-78](shopos-core/src/Modules/CheapestDefaultVariation/Module.php#L65-L78)
- Only 2 toggles. Missing:
  - **Strategy selector**: cheapest / first-in-stock / featured-attribute / random.
  - **Per-product opt-out** (meta box).
  - **Per-category exclude list**.
  - Filter to override `display_price` field selection (some shops use cost+margin).

#### C3. InfiniteScroll — [Module.php:62-68](shopos-core/src/Modules/InfiniteScroll/Module.php#L62-L68)
- Has 3 settings (skeleton_count, max_pages, end_message). Missing:
  - **Trigger mode**: auto-scroll / load-more button / hybrid (auto first 2 pages, button after).
  - **Selector override** (currently hardcoded to `.products`; some themes use different markup).
  - **Per-archive opt-out** (e.g. disable on tag archives).
  - **Analytics events** (toggle to fire `gtag('event', 'page_view')` on each load).
  - **History API integration** toggle.

#### C4. MyAccount
- Zero configurability. Should expose:
  - **Layout mode**: sidebar / tabs / accordion.
  - **Endpoint visibility toggles** (hide downloads/addresses/payment-methods if unused).
  - **Custom endpoint registration** (UI to add menu items pointing to pages).
  - **Avatar upload** (built into many My Account add-ons).

#### C5. ProductFeed — [Module.php:60-78](shopos-core/src/Modules/ProductFeed/Module.php#L60-L78)

  - **Feed format** selector (Google XML, Facebook CSV, Pinterest, etc.).
  - **Field mapping** UI (which WC fields → which feed columns).
  - **Product filter rules** (in-stock only, exclude category, min-price, etc.).
  - **Auth token** (currently public URL — anyone can hit it; could expose pricing).
  - **Multiple feeds** for different channels with different rules.
  - Debounce delay **30s hardcoded** ([Module.php:225](shopos-core/src/Modules/ProductFeed/Module.php#L225)).

#### D2. REST API
- **Zero `register_rest_route()` calls** across the whole suite (verified across all 3 packages).
- Headless / SPA / mobile-app integrations are blind. Restock subscribers, feed status, profiler stats, slider data — all unreachable except via WP-Admin or scraping.

#### D3. CLI commands
- shopos-core has no `WP_CLI::add_command()` calls. shopos-digital gates an include behind `WP_CLI` ([shopos-digital.php:40-42](shopos-digital/shopos-digital.php#L40-L42)) but shopos-core ships no CLI. Ops needs (regenerate feed, list subscribers, fix all variable stock) require admin clicks or DB queries.

#### D4. i18n gaps
- **One confirmed unwrapped string**: [ProductFeed/Server.php:92](shopos-core/src/Modules/ProductFeed/Server.php#L92) — `'Feed not available yet. Please try again in a few seconds.'` is echoed without `__()`.
- Profiler classification names rendered via `str_replace('_', ' ', $r->classification)` in [class-shopos-digital-admin.php:714](shopos-digital/includes/class-shopos-digital-admin.php#L714) — not localized.
- Inline JS in admin.js for shopos-digital likely has hardcoded English strings (not fully audited).
- Restock Notify defaults are Hebrew; English-locale activation requires manually replacing every option default — no locale-aware bootstrapper.

#### D5. theme.json filter coverage
- Plugin ships zero `wp_theme_json_data_theme` / `wp_theme_json_data_default` filters. Modules cannot contribute color/spacing/font presets to the block editor; merchants using block themes won't see ShopOS's tokens in core pickers.

#### D6. Module discovery extensibility
- `shopos_core/modules` filter exists ([Module_Registry.php:81](shopos-core/src/Core/Module_Registry.php#L81)) — third parties can add modules. Good. But no documented public extension point for **third-party settings sections** within the Settings_Hub UI (only `shopos_core/module_page/<id>` per-module).

#### D7. Direct `echo` of HTML
- `Dashboard.php:57, 70, 122, 131, 135, 137, 153-191` — module cards rendered with raw HTML, all text escaped but **no filter on the rendered card markup**. Custom branding / per-card CTA / additional metadata isn't injectable.

#### D8. Logger
- `Logger::log()` is a final class with no extension hook ([Logger.php:16-63](shopos-core/src/Core/Logger.php#L16-L63)). Third parties can't wire logs into Sentry/Datadog/Loggly without forking. Add `shopos_core/logger/entry` filter and `shopos_core/logger/written` action.

#### D9. Frontend asset filtering
- No `apply_filters` on enqueue handles or src URLs. Sites wanting to swap CSS (e.g. white-label) must dequeue and re-register manually.

## 4. Open Questions

1. **Product positioning**: Is ShopOS Core meant as an internal/private suite for ShopOS Digital's clients, or a commercial WooCommerce plugin sold on its own? The recommendations differ — commercial competitors demand wishlist/quick-view/multi-channel feeds; an internal suite can stay opinionated and skip those.

2. **Locale strategy for Restock Notify**: The Hebrew defaults baked into option values ([RestockNotify/Module.php:195-228](shopos-core/src/Modules/RestockNotify/Module.php#L195-L228)) suggest an Israeli market focus. Should English locales auto-substitute defaults on activation, or is the expectation that admins translate manually?

3. **Block editor commitment**: The theme is a Hello Elementor child with no Gutenberg block patterns or block-based templates. Is the Elementor lock-in intentional? If yes, `wp_theme_json_data_*` filtering matters less; if Gutenberg is on the roadmap, design-token unification (B7) becomes urgent.

4. **REST vs admin-AJAX**: Existing endpoints (VariableStockFix bulk audit, ProductFeed admin-post handler) use `wp_ajax_*` and `admin_post_*`. Is there appetite to migrate to REST for consistency, or maintain the AJAX surface for performance?

5. **Backwards-compat ceiling**: VariationSwatches still ships `legacy/` directory ([Module.php:116-135](shopos-core/src/Modules/VariationSwatches/Module.php#L116-L135)) and uses `shopos_vs_*` option keys / shortcode for zero-downtime migration. When (if ever) can the legacy code be retired? This affects how aggressively we can refactor settings to Settings_Hub.

6. **shopos-digital coupling**: The plugin is currently independent of shopos-core. Is the intent to keep them separable forever (so optimization can be sold/used standalone) or is there a future merge?

7. **Per-merchant white-labeling**: Some recommendations (logger filters, asset URL filters, admin skin filter for digital) only matter if you sell to agencies who white-label. Confirm market segment before prioritizing D8/D9.

8. **Production telemetry**: Is opt-in usage analytics (A25) acceptable, or does the privacy posture forbid it? This affects whether we can ship anonymized module-usage tracking to inform future dev priorities.
