#### N-02 (High) — ProductFeed hook namespace

Hooks affected (all in [ProductFeed/Module.php](shopos-core/src/Modules/ProductFeed/Module.php) and [Server.php](shopos-core/src/Modules/ProductFeed/Server.php)):
- cron: `shopos_productfeed_hourly`, `shopos_productfeed_async`
- nonce: `shopos_productfeed_generate_now`
- admin-post action: `admin_post_shopos_productfeed_generate_now`
- query var: `shopos_productfeed`

**Risk:** cron events with the old hook name are already persisted in the `cron` option on every install. A blind rename leaves orphan cron entries that never fire. The query var is referenced in the rewrite rule.

**Migration block:**

1. **One-release-cycle compatibility shim.** For each cron and admin-post action, register handlers on BOTH the old and new hook names; for the cron, also reschedule once on plugin update.

```php
// shopos-core/src/Modules/ProductFeed/Module.php — boot()
add_action( 'shopos_core_product_feed_hourly', array( $this, 'cron_hourly' ) );
add_action( 'shopos_core_product_feed_async',  array( $this, 'cron_async'  ) );
// Deprecated as of 1.9.0, removed in 2.0.0:
add_action( 'shopos_productfeed_hourly', array( $this, 'cron_hourly' ) );
add_action( 'shopos_productfeed_async',  array( $this, 'cron_async'  ) );
```

2. **Migration in `Plugin::on_update()` (or `Migrations::run()`), gated by `shopos_core_db_version`.** Unschedule the old cron and schedule the new:

```php
// Migrations.php — once per install
if ( get_option( 'shopos_core_db_version' ) < 5 ) {
    if ( $ts = wp_next_scheduled( 'shopos_productfeed_hourly' ) ) {
        wp_unschedule_event( $ts, 'shopos_productfeed_hourly' );
    }
    if ( ! wp_next_scheduled( 'shopos_core_product_feed_hourly' ) ) {
        wp_schedule_event( time() + HOUR_IN_SECONDS, 'hourly', 'shopos_core_product_feed_hourly' );
    }
    update_option( 'shopos_core_db_version', 5 );
}
```

3. **Constants** at [Module.php:31-33](shopos-core/src/Modules/ProductFeed/Module.php#L31-L33) and [Server.php:23](shopos-core/src/Modules/ProductFeed/Server.php#L23) updated to canonical names.

4. **Remove shim in 2.0.0** (next major).

`bash tools/release.sh core 1.9.0 "Renamed ProductFeed hooks to canonical shopos_core_product_feed_* with one-release deprecation shim"`

### H-01 — Unsubscribe token uses `wp_generate_password()` ✅ Fixed in 1.9.1
- Module: Restock Notify
- File: [class-shopos-restock-database.php:40](shopos-core/src/Modules/RestockNotify/legacy/includes/class-shopos-restock-database.php#L40)
- Issue: Token generated via `wp_generate_password( 32, false )`. `wp_generate_password()` is documented as not cryptographically secure (uses `wp_rand()`). Public unsubscribe URL is the only thing protecting one customer from another customer's unsubscribe action.
- User impact: A determined attacker who can enumerate subscription IDs *and* token format could potentially unsubscribe other users. Medium-low practical risk (32 chars × 62 alphabet = ~190 bits of entropy regardless of generator weakness), but the right primitive is cheap to use.
- Fix:

```php
// before
$token = wp_generate_password( 32, false );
// after
$token = bin2hex( random_bytes( 16 ) ); // 32 hex chars, CSPRNG
```

CHANGELOG (Security): `Restock Notify unsubscribe tokens now use random_bytes() instead of wp_generate_password().`
